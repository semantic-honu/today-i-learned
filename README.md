####  MCP（Model Context Protocol）とは？
- AIが外部ツールを使うための**共通プロトコル**
- 「AIが関数を呼び出す」ための**標準化された命令文（JSON-RPC）**
- AIが「道具を使える存在」になるための**翻訳者兼エージェント**
####  MCPの構成要素
| コンポーネント   | 役割                                   |
|----------------|--------------------------------------|
| MCPホスト      | AI本体（ChatGPT, Claudeなど）         |
| MCPクライアント | 命令文を構築してサーバーに送る         |
| MCPサーバー    | 実際に関数を実行する（API呼び出し、ファイル操作など） |
####  MCPサーバーの役割
- AIから受け取った命令文をもとに、**関数を実行**
- APIやローカル処理を代行する「AIの手足」
- 画像やファイルも引数として受け取れる
```json
{
  "method": "analyze_ui_image",
  "params": {
    "image": "base64_encoded_image_data",
    "context": "この画面を参考にCLIツールのUIを設計して"
  }
}
```

#### セキュリティTILまとめ

- **セキュリティは人災**  
    多くのサイバー攻撃は、技術的な問題よりも人間の心理や設定ミスなどヒューマンエラーが原因。
- **データ管理の甘さ**  
    企業の個人情報漏洩は、不要な情報を保管し続けることが主な原因。**「データを捨てる勇気」**が重要。
- **攻撃者の戦略**  
    ハッカーは既存の脆弱性や手口を繰り返し利用。多くの企業が基本的な対策を怠っていることを知っている。
- **Googleのセキュリティ**  
    Googleは脆弱性報奨金プログラムで外部研究者を評価し、迅速に修正。業界全体のセキュリティ向上に貢献。
- **SSOトークンのリスク**  
    SSOトークンが盗まれた場合、パスワードリセットは有効だが、全ユーザーのトークン失効は業務停止リスク。
- **侵入後の対策**  
    攻撃者は正規ユーザーになりすますため、NDRやEDRでログイン後の不審な振る舞いを検知することが重要。
- **効果的なセキュリティ啓発**  
    抽象的な警告より、他社の具体的なインシデント事例を共有し、リスクを自分事として捉えさせることが効果的。
    

### 9月11日

- イントラネット停止の具体的な方法  
  - L2/L3スイッチのポートシャットダウン: ネットワーク管理者がスイッチの設定を変更し、特定のポートを物理的に停止させる。すべての機器の接続を個別に切断するのに比べて迅速。  
  - スイッチやルーターの電源オフ: 最も単純かつ確実な方法。ただし、他の重要な通信まで止めてしまうため、影響範囲を考慮する必要あり。  
  - LANケーブルの物理的な切断: 感染が疑われるPCやサーバーのLANケーブルを物理的に抜き、ネットワークから隔離。  
  - ファイアウォールでの通信遮断: ファイアウォールの設定を変更し、社内ネットワーク間の通信をすべて遮断。

- 感染していないと判断する方法  
  - ログ分析: サーバー、PC、ネットワーク機器のログを詳細に分析し、不審な通信や活動がないか確認。  
  - バックアップの検証: 攻撃を受ける前のバックアップデータが安全であることを確認。復旧作業の基礎となるため重要。  
  - フォレンジック調査: 感染が疑われる機器のディスクイメージを取得し、マルウェアの痕跡やバックドアがないか詳細分析。

- FWの設定ミス  
  - IPアドレスの誤設定  
    - 例: 特定の外部協力会社のIPアドレスだけを許可するつもりが、誤って0.0.0.0/0（インターネット上のすべてのIPアドレス）を許可してしまう。  
    - 結果: 外部から誰でもアクセスでき、攻撃者の侵入を許す。  
  - ポート番号の開放ミス  
    - 例: 外部から特定のポート番号だけを開けるべきだったのに、誤って管理用のポート（SSHの22番、RDPの3389番など）も開けてしまう。  
    - 結果: 攻撃者が認証情報の盗難や総当たり攻撃でサーバーに直接アクセスできる脆弱性となる。  
  - ルール設定の順序ミス  
    - 例: 「すべての通信を拒否」ルールを一番下に置くべきなのに、誤って「特定の通信を許可」よりも上に置いてしまう。  
    - 結果: 意図しない通信が許可されてしまう。

- SSOトークン失効の仕組み  
  - セッショントークン (Access Token)  
    - 有効期限が短い。盗まれても期限切れで自動的に無効。  
    - 盗まれた直後から期限切れまでの間は不正利用リスクあり。  
    - 即座に失効させるには、認証サーバーがトークン状態を管理し、ブラックリスト登録が必要。  
  - リフレッシュトークン (Refresh Token)  
    - 有効期限が長く、新しいセッショントークン発行に使われる。  
    - 盗まれると何度でも新しいトークンを発行される。  
    - 失効には認証サーバーがリフレッシュトークンを無効化する必要あり。  
  - 技術的な失効方法  
    - サーバーサイドでの失効管理: トークンをDBで管理し、不正検知時に削除またはブラックリスト化。  
    - ユーザーアカウントの強制ログアウト: 管理者が強制的にログアウトさせ、全トークンを無効化。  
    - パスワードのリセット: パスワード変更時に全トークンを自動的に無効化。  
    - SSOトークンが盗まれた場合は、アカウント強制ログアウトとパスワードリセットが最も確実。

- Cloudflare導入サイトのオリジンIPアドレス特定手法  
  - サブドメインの調査: Cloudflare未設定のサブドメインのDNSレコードからオリジンIPを調査。  
  - DNS履歴の確認: Cloudflare導入前の古いDNSレコードをパブリックDB（Censys, Shodan, DNSDumpsterなど）で検索。  
  - SSL証明書の共通性: Cloudflare以外のサービスで同じ証明書を使っている場合、そのIPがオリジンIPの可能性。  
  - サーバーヘッダー情報の調査: HTTPレスポンスヘッダーからサーバー情報やIP漏洩を確認。  
  - ネットワークエラーメッセージの分析: プロキシ無効時やエラー発生時にIPが表示される場合あり。  
  - これらはOSINT（オープンソースインテリジェンス）の領域であり、完全自動化は困難。手動調査や複数情報源の組み合わせが必要。

#### Spring Boot テスト関連

- `@WebMvcTest(.class)`  
  Spring MVC（Web層）のみを対象にしたテスト用アノテーション。  
  コントローラー単体の動作検証ができ、サービス層やDBはMockで差し替える必要あり。  
  MockMvcが自動で使える（HTTPリクエストのシミュレーションが可能）。

- `@Import(.class)`  
  @WebMvcTestは必要最小限のBeanしか読み込まないため、セキュリティ設定は除外される。  
  コントローラーが@PreAuthorizeやSecurityContextを使う場合、@Importで明示的に設定を追加する。

- `@BeforeEach`（JUnit）  
  各テストメソッドの前に毎回実行される処理を定義するアノテーション。  
  テストごとに新鮮な状態で使える。

- `when(...).thenReturn(...)`（Mockito）  
  モックの振る舞い定義。  
  例：userRepository.findByUsername(...)が呼ばれたとき、Optional.of(...)を返す。

- `@Test`  
  JUnitの基本アノテーション。テストメソッドであることを示す。

- `@WithMockUser(username = ...)`  
  Spring Securityのテスト用アノテーション。  
  ログイン済みユーザーとしてテストを実行する。

- テスト構成例  
  | 層         | テスト内容                                   | 優先度         |
  |------------|--------------------------------------------|----------------|
  | Controller | レスポンス、ビュー名、モデル属性            | 高             |
  | Service    | ロジック分岐、例外処理、依存のモック        | 高             |
  | Repository | クエリ動作確認（@DataJpaTest）              | 中〜低（複雑なら高） |
  | Security   | 認証・認可動作確認（@WithMockUserなど）     | 中             |
  | Integration| 全体の流れ（@SpringBootTest）               | 中（必要に応じて）   |

- `mockMvc.perform(post(...)...)`  
  POSTへの仮想リクエスト。  
  .with(csrf())でCSRFトークンを付与（Spring Security有効時は必須）。  
  .paramでフォーム値指定。  
  .andExpect(status().is3xxRedirection())でリダイレクト確認。  
  .andExpect(redirectedUrl(...))でリダイレクト先確認。

- Controllerテストで確認すべきこと  
  - ビュー名
  - ステータスコード
  - モデル属性
  - リダイレクト先
  - セキュリティ制約

- Controllerテストで確認しないこと  
  - DB保存（Service/Repository層の責任）
  - ビジネスロジックの詳細分岐

#### HTTPステータスコード「3xx」

| ステータス | 名前                | 意味                                 |
|------------|---------------------|--------------------------------------|
| 300        | Multiple Choices    | 複数の選択肢（ほぼ使われない）       |
| 301        | Moved Permanently   | 恒久的に移動（URL変更）              |
| 302        | Found               | 一時的に別の場所（よく使われる）     |
| 303        | See Other           | POST後にGETで別URLへ                 |
| 307        | Temporary Redirect  | 一時的リダイレクト（メソッド保持）   |
| 308        | Permanent Redirect  | 恒久的リダイレクト（メソッド保持）   |

- `.is3xxRedirection()`  
  MockMvcの.andExpect(status().is3xxRedirection())は、ステータスコードが300〜399の範囲であることを確認。  
  より厳密に確認したい場合は`.isFound()`（302）や`.isSeeOther()`（303）なども利用可能。

  /* Y軸に-10px、上へ10px移動 */

  box-shadow: 0 15px 40px rgba(0, 0, 0, 0.12);

- **CSS/Flexbox**
  - FlexboxはCSSのコンテナー定義。
  - `d-flex` はBootstrapのクラスで、要素に `display: flex;` を適用。

- **フラッシュメッセージ**
  - 一度表示されたら消えるメッセージを次ページに渡す用途。
  - ControllerのPostメソッドに `RedirectAttributes` を追加し、`addFlashAttribute(key, value)` を設定。
  - 表示例: `th:if="${key}"` で値があれば `th:text="${value}"` で表示。

- **S/4HANA (SAP最新ERP)**
  - **基盤:** インメモリデータベース「SAP HANA」
  - **特徴:** 超高速処理、リアルタイムデータ活用、シンプル化されたデータモデル
  - **メモリ管理:**  
    - 全データをメモリ保持せず、カラム型ストアによる高圧縮  
    - 頻繁アクセスデータのみメモリ保持でメモリ不足回避
  - **永続性:**  
    - メモリ上データ＋ディスクへのトランザクションログ・セーブポイント書き込みで永続性確保

- **脆弱性 CVE-2025-42957 (S/4HANA ABAPレイヤー)**
  - **概要:** 特定RFCモジュールで入力検証不備により、低権限ユーザーが任意ABAPコード実行可能
  - **根本原因:** 認証後の認可（Authorization）制御の不備
  - **危険性:** ログインした低権限ユーザーがシステム完全制御可能  
    - CVSSスコア: 9.9（非常に高い）

#### Spring Boot セキュリティ

- **CSRF対策**  
  `SecurityConfig.java`でCSRF（クロスサイトリクエストフォージェリ）保護の設定が明記されていないと、意図せず無効化するリスクがある。  
  Spring SecurityはデフォルトでCSRF保護を有効にするが、設定を明示的に記述することで安全性を高められる。
- **設定方法**  
  `securityFilterChain`メソッドにCSRF保護の設定を追加する。

#### Git: 作業の一時退避（git stash）

- **git stash**  
  コミット前の作業中の変更を一時的に保存し、作業ディレクトリをクリーンな状態に戻す。

- **主要コマンド**
  - 変更の一時退避:  
    `git stash`
  - 退避した変更を元に戻す:  
    `git stash pop`
  - スタッシュの内容を確認:  
    `git stash list`
  - 特定のスタッシュを適用:  
    `git stash apply stash@{1}`  
    ※`pop`と異なり、スタッシュから変更を削除しない。

- **ユースケース例**  
  1. 機能開発中のブランチで`git stash`を実行し、変更を一時退避。  
  2. mainブランチ等に`git checkout`で移動。  
  3. バグ修正後、`git add`・`git commit`・`git push`でリモートに反映。  
  4. 元の開発ブランチに戻り、`git stash pop`で変更を復元して開発再開。

* 2の補数表現とオーバーフロー
  * 2の補数表現の範囲は非対称：nビットの2の補数表現の範囲は、-2^{n-1}$から$2^{n-1}-1までである。正の数より負の数が1つ多い。
  * 最上位ビットは符号ビット：2の補数表現では、最も左のビット（最上位ビット）が符号を表す。0は正の数、1は負の数を示す。
  * ビット数によって解釈が変わる：1000という2進数パターンは、4ビット表現では負の数（-8）として扱われるが、5ビット表現では正の数（8）として扱われる。
  * オーバーフローの条件：2つの同じ符号の整数を加算したとき、結果の符号が反転する場合にオーバーフローが発生する。
    * 正の数同士：和が$2^{n-1}$以上になると、最上位ビットが0から1に反転し、負の数として解釈される。
    * 負の数同士：和の絶対値が$2^{n-1}$を超えると、最上位ビットが1から0に反転し、正の数として解釈される。

   * 利便性とセキュリティのトレードオフ: SSOのような便利な機能は、適切な設定（重要操作時の再認証など）を行わないと、かえって大きなリスクとなる。

  - ​CSSが適用されない原因: APサーバーがダウンしているため、APサーバー内で管理されているCSSファイルや、動的に生成されるCSSが配信されず、デザインが崩れる。
  - ​技術スタックの推測: {{...}}構文と、求人情報などの公開情報を組み合わせることで、フロントエンドはVue.jsやAngular、バックエンドはSpring Boot（Java）、インフラはAWSといった技術スタックを推測できる（OSINT）。

 * V8エンジンの脆弱性（CVE-2025-5419）: Google ChromeのJavaScriptエンジン「V8」の最適化処理のバグに起因する脆弱性。
   * ヒープ汚染（Heap Corruption）: 本来のヒープ領域が意図せず上書きされる現象。
   * 原因: V8のStore-Store Eliminationという最適化処理が、特定の条件下で観測点を誤認し、不要なストア（メモリへの書き込み）を削除したため。
   * 結果: あるはずのデータがメモリになく、未初期化のメモリ領域が生じた。
 * 攻撃のステップ（エクスプロイトチェーン）:
   * 情報漏えいプリミティブ: 未初期化のメモリに残ったゴミデータから、メモリ上のアドレス情報を盗み出す。
   * フェイクオブジェクト化: 盗んだアドレス情報を使い、メモリ上に偽のオブジェクトを構築する。
   * 任意アドレス読み書き: 偽のオブジェクトを通じて、メモリ上の任意のアドレスを自由に読み書きする能力を得る。
   * RCE（リモートからのコード実行）: 最終的に、正規のコード領域を悪意のあるコードで上書きし、実行を乗っ取る。
 * セキュリティ技術の回避:
   * ASLR（Address Space Layout Randomization）: 本来、メモリ上のアドレスをランダム化して攻撃を防ぐ技術。情報漏えいプリミティブによって、この技術がバイパスされる。
   * DEP（Data Execution Prevention）/ W^X: データ領域（ヒープ）でコードが実行されるのを防ぐ技術。任意アドレス読み書き能力で、この技術も回避される。
 * ハッカーの手法とAI:
   * ヒープスプレー: 攻撃コードをメモリ上に配置する手法。
   * APT（持続的標的型攻撃）: 特定の標的を狙う高度な攻撃で、このような高コストな脆弱性が使用されやすい。
   * BigInt（n）: 64ビットアドレスのような大きな整数を正確に扱うためのJavaScriptの新しいデータ型。
   * Big Sleep: AIが自律的に脆弱性を発見するGoogleのシステム。AIの進化により、脆弱性発見のスピードが飛躍的に向上している。
 * ITの基礎:
   * 64ビットアーキテクチャ: アドレス空間の基本長が64ビット（8バイト）で、膨大なメモリを扱える。
   * JITコンパイラ: 実行時にコードを機械語に変換し、高速化する仕組み。今回の脆弱性は、そのコンパイラ部分のバグ。
   * 数値の表記: 16進数（0xFFFFnなど）は、長くなる2進数を人間が扱いやすくするための表記法。
